Copilot CLI Harness 구성 문서

현재 이 저장소의 하네스는 Copilot 프로젝트 지침, repo-local skill, agent persona, 로컬 적응 규칙, 내부 reference 자산, 그리고 사용자-facing docs 지식 저장소를 계층적으로 결합하는 구조다. 핵심 의도는 `.agents/skills/`를 skill 본체로 유지하면서, Copilot CLI / Chat이 바로 이해할 수 있는 진입점을 `.github/` 아래에 따로 제공하는 것이다.

프로젝트: Training-Free-SID-Reco repo-local skills: 28개 agent personas: 3개 생성 위치: artifacts/reports/ 기준 시점: 2026-04-08

1. 전체 구조 요약

1) Copilot 진입점

.github/copilot-instructions.md가 Copilot CLI / Chat이 읽기 쉬운 요약 규칙 파일 역할을 한다.

2) 최상위 스키마

AGENTS.md가 저장소 전반의 규칙, 문서 구조, 우선순위를 정의하는 최상위 규칙 파일이다.

3) 로컬 적응 레이어

.harness/reference/local-adaptation.md가 upstream agent-skills를 이 저장소 구조에 맞게 재해석한다.

4) Skill 본체

.agents/skills/ 아래의 SKILL.md들이 실제 실행 가능한 repo-local skill 레이어다.

5) Specialized Agents

.github/agents/ 아래 persona 파일이 리뷰, 테스트, 보안 점검 같은 역할 기반 작업을 돕는다.

6) 검증 게이트

품질 게이트는 uv run pytest, uv run ruff check ., uv run mypy src, uv run sid-reco doctor 기준으로 정렬되어 있다.

7) 관리 대상 문서 레이어

docs/는 하네스 자체가 아니라 하네스가 관리하는 지식 저장소다. 특히 docs/sources/와 docs/wiki/는 docs-manager의 주 대상이다.

8) 내부 reference 레이어

.harness/reference/는 사용자 문서가 아니라 하네스 내부에서 참조하는 support/reference 자산 모음이다. 이름을 분리해 docs/와 역할 충돌을 없앴다.

9) hook archive 레이어

.harness/hooks/는 현재 Copilot CLI의 활성 필수 경로라기보다, 참고/아카이브 성격의 hook 스냅샷을 담는 위치다.

2. 하네스 해석 순서

1

Copilot 프로젝트 지침 로드

.github/copilot-instructions.md에서 프로젝트 성격, 우선 워크플로, 검증 명령, 경계 조건을 빠르게 파악한다.

2

최상위 규칙 확인

AGENTS.md를 읽고 위키/소스/ADR/스키마 관련 우선순위를 확인한다. 이 저장소에서는 이 파일이 가장 높은 규칙 레이어다.

3

로컬 적응 규칙 적용

upstream 예시가 그대로 맞지 않을 때는 .harness/reference/local-adaptation.md를 적용해 경로, 출력 위치, 검증 명령을 이 저장소 방식으로 치환한다.

4

Skill 선택

실제 작업은 .agents/skills/의 skill 이름을 기준으로 라우팅한다. 예: docs-manager, spec, plan, build, test, ship.

5

Persona 보강

리뷰나 검증처럼 역할이 분명한 경우에는 .github/agents/의 persona를 통해 추가적인 관점(코드 리뷰, 테스트, 보안)을 부여한다.

6

docs/ 지식 저장소에 적용

문서 관련 작업이면 하네스는 docs/를 실제 작업 대상로 본다. docs/sources/는 불변 소스, docs/wiki/는 LLM 관리 위키, docs/wiki/INDEX.md는 탐색 인덱스 역할을 한다.

7

내부 reference 읽기

필요 시 하네스는 .harness/reference/ 아래 문서를 읽어 upstream 스냅샷, 로컬 적응 규칙, command draft를 참고한다. 이 레이어는 사용자-facing docs/와 분리되어 있다.

8

저장소 명령과 구조에 맞게 실행

일반 예시 대신 이 저장소의 실제 스택에 맞춘 uv 명령과 Python 구조를 사용한다. 즉, generic 예시보다 repo reality가 우선이다.

4. docs/와의 연결

docs 영역 소유 / 성격 하네스와의 연결 방식
docs/sources/ 사용자 소유, 불변 소스 docs-manager 인제스트의 입력이다. 하네스는 이 경로를 읽을 수는 있지만 직접 수정 대상로 보지 않는다.
docs/wiki/ LLM 소유 위키 문서 하네스의 주 편집 대상이다. 요약, 엔티티, 개념, 비교, 개요, ADR, 로그가 여기 쌓인다.
docs/wiki/INDEX.md 위키 인덱스 쿼리, 린트, 교차참조 유지의 기준 파일이다. 위키 페이지 변경 시 카테고리 README.md와 함께 동기화된다.
docs/README.md 사람용 안내 docs/의 3-레이어 구조를 설명하는 개요 문서다. 실제 스키마 정본은 여전히 AGENTS.md다.
.harness/reference/ 내부 reference 자산 upstream 문서 스냅샷, 로컬 적응 규칙, command draft, persona 스냅샷을 담는 하네스 내부 문서 레이어다. 사용자-facing wiki와는 역할이 다르다.
.harness/hooks/ 내부 hook archive hook 스냅샷과 관련 스크립트를 보관한다. 현재 구조에서는 support/archive 자산으로 취급하며, 사용자-facing 문서 레이어와 직접 섞지 않는다.
핵심 연결점 1

AGENTS.md가 docs/의 구조를 정의한다. 즉, docs/는 독립 문서 폴더가 아니라 하네스 규칙에 종속된 지식 저장소다.

핵심 연결점 2

.github/copilot-instructions.md는 위키/소스/ADR 작업을 만나면 docs-manager를 먼저 쓰도록 라우팅한다.

핵심 연결점 3

.agents/skills/docs-manager/SKILL.md가 실제로 docs/sources/, docs/wiki/, INDEX.md를 읽고 갱신하는 실행 절차를 가진다.

핵심 연결점 4

따라서 docs/는 하네스 제어부가 아니라, 하네스가 우선 규칙을 갖고 운영하는 문서 데이터 레이어다.

핵심 연결점 5

.harness/reference/는 사용자-facing 지식 저장소가 아니라, 하네스가 내부적으로 읽는 reference 레이어다. 그래서 docs/와 이름을 의도적으로 분리했다.

5. 레이어별 역할

레이어 대표 파일/경로 역할
Copilot Entry Layer .github/copilot-instructions.md Copilot이 빠르게 읽는 요약 지침. 프로젝트 개요, 우선 skill, 검증 명령, 경계 조건을 제공한다.
Schema Layer AGENTS.md 이 저장소의 최상위 규칙 문서. 위키 구조, 소스 불변성, 문서 작성 규칙, workflow 우선순위를 정의한다.
Adaptation Layer .harness/reference/local-adaptation.md upstream agent-skills 문서를 이 저장소의 실제 경로와 명령으로 매핑한다.
Execution Layer .agents/skills/*/SKILL.md 실제 작업 단위. 계획, 구현, 테스트, 리뷰, 릴리스, 문서 작업이 skill 이름으로 라우팅된다.
Knowledge Base Layer docs/sources/, docs/wiki/, docs/wiki/INDEX.md 하네스가 실제로 관리하는 문서 대상. 소스 인제스트, 위키 생성, ADR 기록, 인덱스 동기화가 여기서 일어난다.
Persona Layer .github/agents/*.md 특정 역할 기반 작업을 위한 시각 제공. 현재는 code-reviewer, test-engineer, security-auditor가 배치되어 있다.
Evidence Layer README.md, references/, .harness/ 실행 시 참고할 현실 정보, 내부 reference 문서, archive/reference 자산을 제공한다.

7. 라우팅 규칙 예시

문서/위키 관련 요청

docs-manager가 최우선이다. docs/sources/, docs/wiki/, INDEX.md, ADR 작업은 일반 skill보다 이 규칙을 먼저 따른다.

docs 레이어 해석

문서 작업에서 docs/는 부가 폴더가 아니라 별도 지식 저장소다. docs/sources/는 입력, docs/wiki/는 출력 및 유지보수 대상이다.

내부 reference 해석

.harness/reference/는 사용자에게 보여주는 wiki가 아니라 내부 운영 참고 문서다. 하네스는 여기서 로컬 적응 규칙과 upstream 스냅샷을 읽는다.

기능 구현 요청

spec → plan → build → test → code-review-and-quality 흐름으로 연결하기 좋게 구성되어 있다.

버그 수정 요청

test로 failing case를 먼저 만들고, 구현 중 문제는 debugging-and-error-recovery로 넘기도록 설계되어 있다.

릴리스 준비 요청

ship과 shipping-and-launch가 배포 전 점검과 롤아웃 관점을 제공한다.

8. 핵심 규칙 드롭다운

중요한 제어 파일은 아래에서 접었다 펼치며 볼 수 있다. 목적은 “어디가 정본이고, 언제 무엇을 업데이트해야 하는가”를 한눈에 확인하는 것이다.

AGENTS.md — 최상위 규칙 / 스키마 정본

AGENTS.md는 이 저장소 문서 하네스의 최상위 규칙 파일이다. 특히 docs/의 3-레이어 구조와 위키 작성 규칙을 정의한다.

  • 3-레이어 구조 정의: docs/sources/는 불변 원문, docs/wiki/는 LLM 소유 위키, 자신은 스키마 레이어
  • 문서 우선순위: 위키/소스/ADR/인덱스 작업은 항상 docs-manager와 이 규칙을 우선 적용
  • 작성 규칙: YAML frontmatter 필수, 한국어 유지, 사실 기반 작성, source 추적, 양방향 Related 링크 유지
  • 인덱스 규칙: 페이지 생성/삭제/이동 시 docs/wiki/INDEX.md와 카테고리 README.md를 함께 갱신
  • 오퍼레이션 정의: ingest / query / lint의 절차와 출력 형식을 명시
docs/wiki/INDEX.md — 위키 인덱스 정본

INDEX.md는 전체 위키 페이지 목록이며, 문서 하네스가 위키를 탐색하고 동기화할 때 기준으로 삼는 파일이다.

  • 현재 역할: entities, decisions, overviews 등 실제 위키 페이지를 카테고리별 표로 나열
  • 탐색 기준: query와 lint에서 관련 페이지를 찾는 출발점
  • 동기화 규칙: 인제스트 완료, 위키 저장, 린트 자동 수정, 페이지 이동/삭제 시 갱신 필요
  • 함께 움직이는 파일: 각 카테고리 README.md
  • 의미: 실제 파일 시스템과 지식 구조를 연결하는 카탈로그
.github/copilot-instructions.md — Copilot용 진입 규칙

Copilot CLI / Chat이 먼저 읽기 쉬운 프로젝트 요약 규칙이다. 긴 정본 문서 전체를 대신하지 않고, 빠른 라우팅과 기본 경계 조건을 제공한다.

  • 우선 읽기 지시: AGENTS.md, .harness/reference/local-adaptation.md, .agents/skills/
  • 워크플로 라우팅: 문서 작업은 docs-manager, 구현은 build, 테스트 우선 수정은 test
  • 검증 명령: uv run pytest, ruff, mypy, sid-reco doctor
  • 경계: docs/sources/는 수정 금지, docs/wiki/는 한국어 + frontmatter 유지
  • 구조 분리: 사용자-facing 문서는 docs/, 내부 reference 문서는 .harness/reference/에 둔다
.harness/reference/local-adaptation.md — upstream skill의 로컬 해석 규칙

upstream agent-skills 예시를 이 저장소 구조에 맞게 번역하는 파일이다. generic 예시보다 repo reality를 우선하게 만드는 연결점이다.

  • 경로 매핑: 예를 들어 docs/decisions/는 이 저장소에서 docs/wiki/decisions/로 해석
  • 명령 치환: 일반 예시의 npm/build 흐름을 uv 기반 검증 명령으로 전환
  • 우선순위 고정: 위키/ADR/index 작업은 항상 docs-manager 우선
  • 산출물 위치: 아이디어는 ideas/, 계획은 tasks/, 위키/ADR은 docs/wiki/, 리포트는 artifacts/reports/
  • 내부 위치: 현재 이 로컬 적응 규칙 파일 자체는 .harness/reference/ 아래에 둔다
.harness/README.md / .harness/reference/README.md — 내부 하네스 안내

이 두 README는 하네스 내부 자산의 역할 분리를 설명한다. 핵심 목적은 docs/와 .harness/reference/의 의미를 섞지 않는 것이다.

  • .harness/README.md: tool-facing entrypoint와 support 자산의 경계 설명
  • .harness/reference/README.md: imported 문서 스냅샷과 로컬 적응 규칙이 들어 있는 내부 reference 레이어 설명
  • .harness/hooks/: 현재는 support/archive 성격으로 분류

9. 검증 명령이 하네스에 묶이는 방식

이 하네스는 generic JavaScript 예시를 그대로 쓰지 않고, 저장소의 Python + uv 현실에 맞춰 품질 게이트를 고정했다.

uv sync --all-groups
uv run pytest
uv run ruff check .
uv run mypy src
uv run sid-reco doctor

추가로 MLX / taxonomy 관련 명령도 도메인별 검증 대상으로 연결된다.

10. 이 구성이 Copilot CLI에 유리한 이유

  • 진입점 분리: Copilot이 읽기 쉬운 `.github/`와 실행 skill 본체인 `.agents/skills/`가 역할별로 분리되어 있다.
  • 우선순위 명확화: `AGENTS.md`와 `local-adaptation.md`가 충돌 해소 규칙을 명시한다.
  • 명령 현실화: upstream 예시 대신 `uv` 기반 실제 검증 명령으로 정렬되어 있다.
  • 문서/코드 워크플로 분리: 위키 작업은 `docs-manager`, 구현/테스트는 imported skill로 라우팅된다.
  • docs 레이어 분리: 제어용 하네스 파일과 실제 문서 저장소 docs/가 분리되어 있어, 지식 관리 대상이 명확하다.
  • 내부 reference 명확화: .harness/reference/라는 이름으로 내부 문서를 분리해, 사용자-facing wiki와 운영 reference를 혼동하지 않게 했다.
  • 역할 보강: review / test / security persona를 `.github/agents/`에 따로 둬서 review성 작업을 분명하게 만들었다.

11. 현재 상태 평가

항목 상태 메모
Copilot instructions 구성됨 .github/copilot-instructions.md 존재
Repo-local skills 구성됨 .agents/skills/에 28개 skill 배치
Specialized agents 구성됨 .github/agents/에 3개 persona 배치
Top-level rules 구성됨 AGENTS.md가 최상위 규칙
Upstream adaptation 구성됨 .harness/reference/local-adaptation.md 존재
Managed docs layer 구성됨 docs/sources/, docs/wiki/, docs/wiki/INDEX.md가 문서 하네스의 대상 경로로 연결됨
Internal reference layer 구성됨 .harness/reference/가 하네스 내부 문서 레이어로 분리됨
Hook archive layer 구성됨 .harness/hooks/가 support/archive 자산 경로로 분리됨
결론적으로, 현재 하네스는 Copilot CLI가 바로 해석 가능한 진입 파일과 repo-local skill 실행 레이어, 실제 문서 대상인 docs/ 지식 저장소, 그리고 내부 support/reference 레이어인 .harness/를 함께 갖춘 구조다. 즉, "skill 본체는 `.agents/skills/`", "Copilot 친화적 진입점은 `.github/`", "사용자-facing 문서는 `docs/`", "내부 운영 reference는 `.harness/reference/`"라는 다층 연결 구조로 설계되어 있다.
