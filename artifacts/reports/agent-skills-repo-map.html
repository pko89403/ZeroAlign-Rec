1. 저장소 구조는 어떻게 읽어야 하나

이 리포는 “실행 코드”보다 “운영 규칙” 중심이다. 핵심 자산은 스킬, 에이전트 페르소나, 참조 체크리스트, Claude용 명령 래퍼, 도구별 설치 가이드다. 

agent-skills/
├── skills/                    19개 핵심 워크플로 스킬
│   ├── idea-refine/
│   ├── spec-driven-development/
│   ├── planning-and-task-breakdown/
│   ├── incremental-implementation/
│   ├── test-driven-development/
│   ├── code-review-and-quality/
│   └── ...
├── agents/                    코드리뷰/테스트/보안 페르소나
├── references/                테스트·성능·보안·접근성 체크리스트
├── .claude/commands/          Claude Code slash commands
├── docs/                      Cursor/Gemini/Copilot/Codex 가이드
├── hooks/                     세션 라이프사이클용 훅
├── AGENTS.md                  기여/패키징 규칙
└── README.md                  전체 개념도와 카탈로그

빠른 해석

  • skills/가 본체다. 각 디렉터리의 SKILL.md가 실제 재사용 단위다.
  • agents/는 별도 모델이 아니라 리뷰 관점 프롬프트에 가깝다.
  • references/는 필요할 때만 읽는 보조 체크리스트다.
  • .claude/commands/는 Claude Code에서 스킬을 쉽게 호출하기 위한 래퍼다.
  • docs/getting-started.md는 Codex 같은 비-Claude 환경을 위한 범용 사용법이다.

2. 스킬 한 개는 어떤 모양인가

이식 단위는 전체 저장소가 아니라 “스킬 파일 하나”다. 문서상으로도 SKILL.md를 시스템 프롬프트, 규칙 파일, 대화 컨텍스트에 넣는 방식을 권장한다. 

skills/
  skill-name/
    SKILL.md
    supporting-file.md (optional)

SKILL.md anatomy
├── YAML frontmatter
│   ├── name
│   └── description
├── Overview
├── When to Use
├── Core Process / Workflow
├── Specific Techniques / Patterns
├── Common Rationalizations
├── Red Flags
└── Verification

이 포맷이 중요한 이유

  • 지식 문서가 아니라 “행동 절차”를 담는다.
  • 설명보다 검증과 반-합리화 항목이 강하다.
  • 에이전트가 요약만 읽고 끝내지 않도록 description은 트리거만 담는다.
  • 큰 참고 자료는 supporting file로 분리하는 progressive disclosure 구조다.

4. Codex에서 무엇을 그대로 쓸 수 있고, 무엇은 바꿔야 하나

문서상 근거는 분명하다. docs/getting-started.md는 “Markdown instructions를 받아들이는 어떤 AI coding agent에도 동작한다”고 말한다. 다만 Claude Code 통합 요소까지 그대로 작동한다는 뜻은 아니다.

그대로 사용 가능
  • skills/*/SKILL.md 본문
  • agents/*.md 페르소나 문서
  • references/*.md 체크리스트
  • README의 단계별 분류 체계
  • using-agent-skills 같은 메타-스킬 라우팅 방식
우리 환경에 맞게 변환 필요
  • .claude/commands/* → Codex용 repo-local 규칙 문서
  • 검증 명령 예시 → uv run pytest, uv run ruff check . 등으로 치환
  • UI/API 예시 → 우리 프로젝트의 CLI/MLX/taxonomy 파이프라인 중심으로 재서술
  • 리뷰 기준 → Python, 로컬 데이터 파이프라인, 문서 위키 운영에 맞게 보정
직접 이식하면 안 되는 것
  • Claude Marketplace 설치 절차
  • Claude 전용 slash command UX를 그대로 전제하는 설명
  • 현재 저장소와 무관한 웹앱/프론트엔드 중심 검증 문구
  • 우리 AGENTS.md의 3-레이어 지식 저장소 규약을 덮어쓰는 방식

5. Training-Free-SID-Reco로 옮길 때의 배치 지도

현재 저장소에는 Codex용 로컬 skill 체계를 둘 수 있다. 즉 agent-skills는 새 체계가 아니라, 기존 체계 옆에 들어올 “확장 레이어”로 배치하는 편이 안전하다.

레이어 A · 유지할 것
  • AGENTS.md의 3-레이어 지식 저장소 정의
  • docs/wiki/ 페이지 타입, 인제스트/쿼리/린트 규약
  • docs-manager 로컬 스킬과 `AGENTS.md` 기반 문서 규칙
레이어 B · 새로 넣을 것
  • .agents/skills/ 아래에 phase별 스킬 추가
  • 예: spec-driven-development, test-driven-development, code-review-and-quality
  • 짧은 `/spec`, `/plan` 같은 호출은 wrapper skill로 제공
레이어 C · 우리 식으로 재작성할 것
  • 검증 명령: uv run pytest, uv run ruff check ., uv run mypy src
  • 도메인 예시: Food.com 전처리, taxonomy step1, taxonomy dictionary, MLX 추론
  • 문서화: ADR 생성은 docs/wiki/decisions/ 규약에 맞춤
레이어 D · 나중에 볼 것
  • frontend-ui-engineering처럼 현재 repo와 직접 관련이 낮은 스킬
  • 브라우저 전용 검증 체계
  • 세션 hook 자동화

6. 전체 이식을 원할 때의 현실적인 순서

“전체 이식”은 19개 디렉터리를 복사하는 작업이 아니다. 현재 저장소 문맥으로 재검증하며 단계적으로 옮겨야 유지보수 가능한 자산이 된다.

기반 정리
  • .agents/skills/를 기준으로 skill 디렉터리 구조를 고정한다.
  • AGENTS.md에 “문서 스키마”와 “개발 워크플로”의 책임 경계를 명시한다.
핵심 스킬 포팅
  • Define, Build, Review 축의 핵심 스킬부터 한국어/로컬 검증 명령으로 재작성한다.
  • 각 스킬에 우리 코드베이스 예시를 넣는다.
명령 레이어 연결
  • 필요한 짧은 호출은 wrapper skill로 추가한다.
  • docs-manager와 일반 구현 스킬의 우선순위를 분리한다.
실전 검증
  • 버그 수정, CLI 변경, 위키 업데이트 같은 실제 작업에 적용한다.
  • 불필요한 단계와 중복 규칙을 줄여 최종 체계로 고정한다.
